<objective>
Add a workspace index page so returning users can navigate to their workspaces after login.

Currently, post-login redirects to /workspaces/new regardless of whether the user already
has workspaces. There is no /workspaces index and no listing UI. This plan creates:
1. getUserWorkspaces — a data helper that reads from the non-RLS better-auth tables
   (organization + member) to return the session user's workspace memberships.
2. /workspaces page — a Server Component that renders the list with links; falls back to
   showing a "Create your first workspace" prompt when the list is empty.
3. Post-login redirect updated from /workspaces/new → /workspaces.

Purpose: Close UAT gap Test 10 — returning users can reach their workspaces without
memorizing URLs.
Output: listing.ts, /workspaces/page.tsx, updated login page, new tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → /workspaces | GET request; session must be present and email-verified before any DB access |
| requireVerifiedUser() → DB | Session validated against the better-auth session table server-side |
| getUserWorkspaces(userId) → DB | userId sourced from session, never from URL/query param |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-08-01 | Spoofing | /workspaces page | mitigate | requireVerifiedUser() validates the session server-side before calling getUserWorkspaces; an unauthenticated request is redirected to /login — the userId cannot be forged by the browser |
| T-02-08-02 | Information Disclosure | getUserWorkspaces enumeration | mitigate | getUserWorkspaces(userId) filters by userId from the session — there is no URL parameter, query string, or client-supplied input that could enumerate another user's workspaces; the query is WHERE userId = session.user.id |
| T-02-08-03 | Information Disclosure | RLS bypass via workspace listing | mitigate | getUserWorkspaces reads from organization/member (non-RLS tables) using the session userId as the filter — it does NOT read from the RLS-protected workspace/workspace_member tables, so there is no SET LOCAL bypass needed and no risk of accidentally listing cross-tenant rows |
| T-02-08-04 | Elevation of Privilege | Post-login redirect | accept | Changing redirect from /workspaces/new to /workspaces is a usability fix; the workspaces page still calls requireVerifiedUser() which enforces auth — no elevation possible |
</threat_model>
