<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → AcceptButton | User triggers accept; invitationId flows from server-rendered prop, not URL param |
| AcceptButton → acceptInvitationAction | Server Action boundary; invitationId is the only input |
| acceptInvitationAction → DB | Email-match guard fires server-side before any membership write |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07-01 | Spoofing | AcceptButton → acceptInvitationAction | mitigate | invitationId is set from server-rendered params (await params), never from a user-supplied form field or query string; acceptInvitationAction re-looks up the invitation row server-side — the client cannot forge workspaceId or role |
| T-02-07-02 | Information Disclosure | Error message content | mitigate | Error message must say "This invitation was issued to a different email address." — must NOT reveal whether the invited address is or is not a registered account (prevents user enumeration via the invite page) |
| T-02-07-03 | Tampering | Client-side error state | accept | AcceptButton holds only a display string in useState; tampering with it has no server-side effect — the server re-validates on every acceptInvitationAction call |
| T-02-07-04 | Denial of Service | Repeated accept clicks | accept | Next.js Server Actions are POST-only (CR-03 already enforces this); useTransition disables the button during in-flight calls, limiting rapid re-submission; no rate-limit added in v1 |
</threat_model>
