From src/engine/renderer.ts (brand scope key resolution — CRITICAL):
Brand tokens in templates use names like brand.logo, brand.primary_color, brand.whatsapp.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→Server Action | LP values, names, template IDs come from form POST — untrusted |
| Server Action→DB | workspaceId must come from server session, never client payload |
| Server Action→S3 | S3 credentials are server-side only; presigned URLs use SigV4 |
| DB JSON columns→engine | LandingPage.values and markupSnapshot loaded from DB and passed to render() |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01-01 | Elevation of Privilege | TenantLpHelpers.findById | mitigate | Every findById/update/delete includes workspaceId filter in where clause; RLS backstop via SET LOCAL ensures cross-workspace lookup returns null even if app-level filter is bypassed |
| T-04-01-02 | Tampering | LandingPage.values (jsonb) | mitigate | Values are validated by deriveZodSchema (runtime Zod resolver) before storage; engine applies per-type sanitization (sanitizeRichText, sanitizeUrl, sanitizeCssColor) during render |
| T-04-01-03 | Information Disclosure | S3 credentials in env | mitigate | S3_ACCESS_KEY_ID / S3_SECRET_ACCESS_KEY are server-side env vars only; never included in client bundle; presigned URLs generated server-side |
| T-04-01-04 | Tampering | brand scope in renderLp | mitigate | Brand values fetched from DB server-side via withTenantDb; never from client input; brandScope keys mapped explicitly (logo, primary_color, whatsapp) — no arbitrary client key injection into render scope |
| T-04-01-05 | Elevation of Privilege | Client-supplied workspaceId | mitigate | TenantClient interface never accepts workspaceId from client; always derived from requireWorkspace() / requireWorkspaceRole() which reads the server session |
</threat_model>
