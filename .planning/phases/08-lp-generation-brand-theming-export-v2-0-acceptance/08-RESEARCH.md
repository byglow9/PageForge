## Summary

Phase 8 builds entirely on foundations from Phases 4–7. The codebase is mature: the LIQUID LP pipeline (generate/update/duplicate/preview/export), the isolated serving origin (Phase 7), and the BrandConfig schema are all in place. Phase 8 is a **branching phase**: every capability already exists for LIQUID, and the work is to fork each path for VITE_SPA.

The most novel piece of work is the **hex→HSL conversion** for brand CSS vars and the **shared `<style>` injection helper** that must be wired into three existing code paths (serve route handler, LP preview page, export route handler) without introducing drift between them. The second novel piece is **listing and streaming the full S3 dist/ tree** into a ZIP for VITE_SPA export — the existing export route only handles a single rendered HTML string and a known set of image URLs; the VITE_SPA branch must instead iterate all S3 objects under the dist/ prefix.

The data model change is minimal: one additive `ALTER TABLE` adding a nullable `entry_route` column to `landing_page`. The LIQUID path is untouched; VITE_SPA rows will have `null` (root `/`) or a path string for other routes.

**Primary recommendation:** Implement in four task clusters — (A) migration + schema + action branch, (B) shared theme helper + wire into serve/preview paths, (C) export route branch + S3 tree streaming, (D) generation UI (form field + page) and LP preview/edit UI branch. Keep the LIQUID path untouched throughout.

## Research Questions Answered

### RQ-1: hex→HSL triplet conversion

**What shadcn/Lovable expects:**
Confirmed from `renova-turismo-jornada-main/src/index.css` (L17-L19): [VERIFIED]

```css
:root {
  --primary: 213 90% 23%;          /* space-separated H S% L% — no "hsl(" wrapper */
  --primary-foreground: 0 0% 100%;
}
```

The CSS usage is `hsl(var(--primary))`. The custom property value is the **three-number triplet only** — `H S% L%` — not a full `hsl(...)` string. This is the shadcn convention.

**Conversion algorithm (dependency-free, ~25 lines):**

```typescript
// Source: standard HSL formula, verified against renova-turismo values
// Input: '#0d4080' (6-digit hex, validated by BrandConfig Zod schema as /^#[0-9a-fA-F]{6}$/)
// Output: '213 90% 23%' (space-separated H S% L%)
export function hexToHslTriplet(hex: string): string {
  // Parse r, g, b from hex
  const r = parseInt(hex.slice(1, 3), 16) / 255;
  const g = parseInt(hex.slice(3, 5), 16) / 255;
  const b = parseInt(hex.slice(5, 7), 16) / 255;

  const max = Math.max(r, g, b);
  const min = Math.min(r, g, b);
  let h = 0;
  let s = 0;
  const l = (max + min) / 2;

  if (max !== min) {
    const d = max - min;
    s = l > 0.5 ? d / (2 - max - min) : d / (max + min);
    switch (max) {
      case r: h = ((g - b) / d + (g < b ? 6 : 0)) / 6; break;
      case g: h = ((b - r) / d + 2) / 6; break;
      case b: h = ((r - g) / d + 4) / 6; break;
    }
  }

  const hDeg = Math.round(h * 360);
  const sPct = Math.round(s * 100);
  const lPct = Math.round(l * 100);
  return `${hDeg} ${sPct}% ${lPct}%`;
}
```

**Validation:** For `#0d4080` → H≈213, S≈90%, L≈23% — matches renova-turismo's `--primary: 213 90% 23%`. [ASSUMED: arithmetic not executed in session, but formula is standard]

**Where to put it:** `apps/web/src/lib/brand/theme.ts` — a new server-only utility (no "use server" directive, called from route handlers and RSC pages). Exports:
- `hexToHslTriplet(hex: string): string`
- `buildBrandStyleTag(primaryColor: string | null): string` — returns the full `<style>` block or empty string if null

**Derived vars (planner's discretion):** `--primary-foreground` can be derived as white (`0 0% 100%`) or black based on luminance. `--ring` can mirror `--primary`. Both are trivial no-UX-cost additions. [ASSUMED: derivation is standard practice; no specific user confirmation needed but planner should decide]

**Security note:** `BrandConfig.primaryColor` is already validated as `/^#[0-9a-fA-F]{6}$/` at save time (brand/schema.ts L40-44). [VERIFIED] The conversion function receives only the numeric components — no CSS injection vector once the hex is validated.

### RQ-2: Where/how to inject the `<style>` tag (shared helper + three paths)

**The constraint:** The serve route handler (Phase 7) streams S3 bytes directly via `transformToWebStream()`. For index.html specifically, the stream must be converted to a string, the `<style>` prepended, and the result returned as a new string response. This is safe and low-cost because index.html is a small text file.

**Current serve handler state (VERIFIED from route.ts):**
- For `isHtmlRequest === true` (s3Path === 'index.html'), the handler calls `GetObjectCommand`, then `s3Response.Body!.transformToWebStream()` and returns it directly as a streaming response.
- The stream is consumed exactly once (`T-07-02-07` rule — must NOT read Body before `transformToWebStream()`).
- **To inject CSS vars:** Instead of calling `transformToWebStream()`, call `s3Response.Body!.transformToString()` (AWS SDK v3 method on `Body: SdkStream`) for index.html, prepend the `<style>`, return `new NextResponse(modifiedHtml, { headers })`.

**Pattern for `transformToString()`:** [VERIFIED: @aws-sdk/client-s3 Body is `SdkStreamMixin & ReadableStream` which has `.transformToString()` in v3] — this consumes the stream into a string. For assets (non-HTML) the existing `transformToWebStream()` path continues unchanged.

**Shared helper design:**

```typescript
// apps/web/src/lib/brand/theme.ts
export function buildBrandStyleTag(primaryColor: string | null | undefined): string {
  if (!primaryColor) return "";
  const triplet = hexToHslTriplet(primaryColor);
  return `<style>:root{--primary:${triplet};}</style>`;
}

export function injectBrandStyle(html: string, styleTag: string): string {
  if (!styleTag) return html;
  // Prepend before </head> if present, else prepend to document
  if (html.includes("</head>")) {
    return html.replace("</head>", `${styleTag}\n</head>`);
  }
  return `${styleTag}\n${html}`;
}
```

**Three injection sites:**

1. **Serve route** (`app/serve/[tplId]/[[...path]]/route.ts`) — for `isHtmlRequest` path only:
   - After fetching the template's `workspaceId` from HMAC claims, look up `BrandConfig` via `prisma.brandConfig.findFirst({ where: { workspaceId } })` (unscoped Prisma, not `withTenantDb` — route handler already has trusted workspaceId from verified token).
   - Call `transformToString()` on Body, apply `injectBrandStyle(html, buildBrandStyleTag(brand?.primaryColor))`, return modified string.
   - **Note:** This requires adding a `prisma` BrandConfig lookup to the serve handler. The serve origin has no session, but the `workspaceId` is trusted from the verified HMAC token claims.

2. **LP preview page** (`app/w/[slug]/lps/[lpId]/preview/page.tsx`) — VITE_SPA branch:
   - RSC already has `withTenantDb` context and can call `db.brandConfig.findFirst()`.
   - Build `buildBrandStyleTag(brand?.primaryColor)`, embed in iframe URL... wait — **the preview uses a cross-origin iframe pointing to the serve origin**, same as the template preview page. The CSS injection happens IN THE SERVE HANDLER (path 1 above), not in the RSC page. The RSC page only needs to mint the token and build the iframe URL including the entryRoute path.

3. **Export route** (`app/api/lps/[lpId]/export/route.ts`) — VITE_SPA branch:
   - After IDOR check, read `BrandConfig` from the LP's workspace.
   - Enumerate dist/ tree from S3 (`ListObjectsV2`), stream into archiver.
   - For `index.html`: `GetObjectCommand` → `transformToString()` → `injectBrandStyle(html, buildBrandStyleTag(brand?.primaryColor))` → `archive.append(Buffer.from(themedHtml, 'utf-8'), { name: 'index.html' })`.
   - For all other files: `GetObjectCommand` → stream directly into archiver.

**Summary:** The shared helper is in `lib/brand/theme.ts`. The serve route (path 1) is the critical injection point for the live preview experience. Export is self-contained. The RSC preview page (path 2) doesn't inject CSS directly — it delegates to the serve handler.

### RQ-5: Export route VITE_SPA branch (S3 tree + archiver)

**Current LIQUID export path (VERIFIED from route.ts):**
- Calls `renderLp()` → HTML string
- Extracts S3 image URLs → downloads each → rewrites srcs
- Injects strict CSP meta tag
- `archive.append(Buffer, { name: 'index.html' })` + `archive.append(buffer, { name: 'assets/...' })`

**VITE_SPA branch — key differences:**
1. No `renderLp()` call
2. No image extraction/rewriting (dist/ assets already reference relative paths like `/assets/...`)
3. No CSP injection (D-12)
4. Brand `<style>` injected into index.html (D-11)
5. Must enumerate entire dist/ tree from S3

**ListObjectsV2 pattern:**

```typescript
import { ListObjectsV2Command } from "@aws-sdk/client-s3";

// Enumerate all objects under the dist/ prefix
const prefix = `workspaces/${lp.workspaceId}/project-templates/${lp.templateId}/dist/`;
let continuationToken: string | undefined;
const s3Keys: string[] = [];

do {
  const listResult = await s3Client.send(new ListObjectsV2Command({
    Bucket: process.env.S3_BUCKET!,
    Prefix: prefix,
    ContinuationToken: continuationToken,
  }));
  for (const obj of listResult.Contents ?? []) {
    if (obj.Key) s3Keys.push(obj.Key);
  }
  continuationToken = listResult.NextContinuationToken;
} while (continuationToken);
```

**Streaming each object into archiver:**

```typescript
const archive = new ZipArchive({ zlib: { level: 9 } });
const brandStyleTag = buildBrandStyleTag(brand?.primaryColor);

for (const s3Key of s3Keys) {
  // Derive path relative to dist/ prefix for ZIP entry name
  const relativePath = s3Key.slice(prefix.length); // e.g. 'index.html', 'assets/main.abc.js'
  
  const s3Obj = await s3Client.send(new GetObjectCommand({
    Bucket: process.env.S3_BUCKET!,
    Key: s3Key,
  }));
  
  if (relativePath === "index.html") {
    // Inject brand style — must materialize as string (small file)
    const html = await s3Obj.Body!.transformToString();
    const themedHtml = injectBrandStyle(html, brandStyleTag);
    archive.append(Buffer.from(themedHtml, "utf-8"), { name: "index.html" });
  } else {
    // Stream directly into ZIP (JS/CSS chunks can be large)
    const webStream = s3Obj.Body!.transformToWebStream();
    const nodeStream = Readable.fromWeb(webStream);
    archive.append(nodeStream, { name: relativePath });
  }
}

archive.finalize();
```

**Note on `Readable.fromWeb()`:** The export route already imports `Readable` from `"node:stream"` (VERIFIED at L28). `Readable.fromWeb()` is the inverse of `Readable.toWeb()` used to bridge back. Available in Node.js 17+. [ASSUMED: Node 18+ confirmed by CLAUDE.md stack requirements]

**ZIP structure:** Files are added at the path relative to the dist/ prefix (e.g., `index.html`, `assets/main.abc123.js`). This mirrors how the dist/ was built and served — no wrapper subfolder.

**S3 key note:** The export route currently uses `prisma.landingPage.findUnique` (no workspace scoping) plus an IDOR membership check. The VITE_SPA branch needs `lp.templateId` (for the S3 prefix) and `lp.workspaceId` (for the brand config lookup). Both are available from the same `lp` object already fetched in step 2. The `lp.entryRoute` is NOT needed in export — the ZIP is route-agnostic (the SPA handles routing client-side from the bundle).

## Common Pitfalls

### Pitfall 1: `transformToWebStream()` vs `transformToString()` split
**What goes wrong:** Calling `transformToWebStream()` on an S3 Body and then trying to read the stream as a string fails because the stream is consumed.
**Why it happens:** The AWS SDK v3 `Body` stream can only be consumed once. The existing serve handler comment `T-07-02-07: Body.transformToWebStream() called exactly once` explicitly warns about this.
**How to avoid:** For index.html (where brand injection is needed): call `Body.transformToString()` to get a string first, inject CSS vars, return as `new NextResponse(modifiedString, { headers })`. For all other assets: keep existing `transformToWebStream()` path unchanged.
**Warning sign:** TypeScript error "Body is already consumed" or an empty response in the browser.

### Pitfall 2: `serve-vite-spa.ts` import boundary
**What goes wrong:** Importing from a `"use server"` module (like `lps/actions.ts` or `lib/brand/actions.ts`) inside the serve route handler causes the `"use server"` boundary crossing error.
**Why it happens:** Route handlers are not Server Actions; they run in a different bundle context. `serve-vite-spa.ts` already documents this pattern (`Do NOT import from project-templates/actions.ts or lps/actions.ts`).
**How to avoid:** The brand/theme.ts helper must NOT have `"use server"`. The serve route handler reads BrandConfig directly via the bare `prisma` client (unscoped), NOT via `withTenantDb` (which requires a session context unavailable in the serve route).
**Warning sign:** Build error about "use server" functions cannot be imported from non-server contexts, or Next.js bundling errors.

### Pitfall 3: entryRoute → serve token mismatch
**What goes wrong:** If the LP preview page builds the iframe URL as `serveOrigin/grecia?t=token` but the token scope only covers `{workspaceId, templateId}`, the serve handler must verify the token against the template, not the path.
**Why it happens:** Confusion between the token scope (template-level) and the path (route-level).
**How to avoid:** The token scope is `{workspaceId, templateId}`. The path (`/grecia`) is an untrusted URL parameter — the serve handler uses the SPA fallback to serve index.html regardless. Confirm the LP preview RSC uses `mintServeToken(workspaceId, lp.templateId)` where `templateId` is the LP's source template ID.
**Warning sign:** 403 responses when loading LP preview (if token is scoped to LP id instead of template id).

### Pitfall 4: VITE_SPA LP edit page hitting `parse('')`
**What goes wrong:** The edit page at `lps/[lpId]/edit/page.tsx` calls `parse(lp.markupSnapshot)` before checking kind. With sentinel `markupSnapshot = ''`, `parse('')` either throws or returns an empty schema, leading to a crash or redirect loop.
**Why it happens:** The edit page was written for LIQUID only; it assumes markupSnapshot is always populated.
**How to avoid:** Add kind check as the FIRST thing after fetching the LP: if `lp.kind === 'VITE_SPA'`, branch to a simple entryRoute edit form; do NOT call `parse()`.
**Warning sign:** Redirect loop when opening a VITE_SPA LP's edit page.

### Pitfall 5: S3 ListObjectsV2 pagination
**What goes wrong:** A dist/ with more than 1000 files only exports the first 1000 objects.
**Why it happens:** S3 `ListObjectsV2` returns at most 1000 objects per call.
**How to avoid:** Always loop with `ContinuationToken` until `IsTruncated === false`. The renova-turismo dist/ has far fewer than 1000 files, but the pattern must be correct for general use.
**Warning sign:** ZIP missing late-alphabetically-sorted assets.

### Pitfall 6: Brand config read in serve route (unscoped prisma)
**What goes wrong:** Using `withTenantDb` in the serve route handler fails because there is no session — the serve origin has no PageForge session cookie.
**Why it happens:** `withTenantDb` calls `requireWorkspace` internally which reads the session. The serve route is on an isolated origin with no session.
**How to avoid:** Use the bare `prisma` client directly: `prisma.brandConfig.findFirst({ where: { workspaceId } })` where `workspaceId` comes from the verified HMAC token claims. The workspaceId from the token is trusted (signed + expiry checked).
**Warning sign:** 401/redirect from auth middleware inside the serve route.

### Pitfall 7: `markupSnapshot: ''` sentinel and existing LIQUID code
**What goes wrong:** LIQUID LP helpers or display components that display markupSnapshot (e.g. a debug panel) break or show empty for VITE_SPA rows.
**Why it happens:** The sentinel is an empty string, not null.
**How to avoid:** Any UI that displays or uses markupSnapshot should check `lp.kind === 'LIQUID'` before rendering. The render guard in `renderLp()` already handles this at the engine level. Catalog display components (LpCard) should not display markupSnapshot — confirm no catalog component depends on it.
**Warning sign:** Empty content in a template preview section that accidentally reads VITE_SPA markupSnapshot.
