<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| RSC → Client Component | canEdit is server-resolved via can(ctx.role,'lp','update'); never derived from client-side input |
| postMessage: iframe → parent | All incoming messages validated against event.origin === serveOrigin (serve subdomain origin) before any state update |
| postMessage: parent → iframe | sendToIframe uses serveOrigin as targetOrigin (never '*') |
| Client Component → Server Action | updateLpAction independently calls requireWorkspaceRole(["owner","admin","editor"]) — dual gate |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-03-01 | Elevation of Privilege | Viewer invoking updateLpAction | mitigate | RSC-level: canEdit=false → no 'Editar' button rendered → ?edit=1 never in iframeSrc (UI gate). Server Action: requireWorkspaceRole(["owner","admin","editor"]) throws/redirects for viewer (authoritative gate). Independent; either alone blocks the attack vector. |
| T-10-03-02 | Spoofing / Tampering | Rogue postMessage from non-serve origin | mitigate | event.origin === serveOrigin check in message handler (RESEARCH Unknown 1 allowlist). Messages from any other origin are silently ignored. serveOrigin is computed server-side in the RSC and passed as a prop; it is not user-supplied. |
| T-10-03-03 | Tampering | Crafted PENDING_EDITS payload with injected override data | mitigate | PENDING_EDITS message carries PfOverride[]; each override is validated by SaveViteSpaOverridesSchema (Zod) inside updateLpAction before DB write.
