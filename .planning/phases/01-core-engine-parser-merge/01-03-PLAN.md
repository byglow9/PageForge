<threat_model>
## Trust Boundaries

| Boundary | Descrição |
|----------|-----------|
| Valor do usuário → sanitizeRichText | HTML raw de campos richtext — potencialmente hostil; a lib sanitize-html é o gatekeeper |
| URL de usuário → sanitizeUrl | href/src de campos button/image — pode ser javascript:, data:, vbscript: |
| CSS de usuário → sanitizeCssColor | Valor de propriedade CSS — pode ser expression(), url(javascript:), import() |
| scope Liquid → HTML output | Após pré-processamento, valores chegam ao LiquidJS como dados seguros — outputEscape é a última camada |

## STRIDE Threat Register (Plano 03 — Sanitizadores + Corpus de Segurança)

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | XSS Stored via rich-text (script, onerror, on*) | mitigate | sanitize-html v2.17.4 com allowlist estrita (D-11): apenas p,strong,em,b,i,ul,ol,li,a,br; `disallowedTagsMode:'discard'`; `parseStyleAttributes:false`; validado por 10 payloads × richtext no corpus |
| T-03-02 | Tampering | XSS via javascript: URL em href (campo button) | mitigate | sanitizeUrl() rejeita qualquer URL não-matching /^(https?:\/\/\|mailto:\|tel:)/i → retorna '#'; LiquidJS outputEscape escapa o '#' em atributo; validado por payload P4 no corpus |
| T-03-03 | Tampering | XSS via data: URI em href/src | mitigate | sanitizeUrl() rejeita data: (não está na allowlist de schemes); validado por payload P9 no corpus |
| T-03-04 | Tampering | CSS Injection via expression() (IE legacy, edge cases) | mitigate | sanitizeCssColor() rejeita qualquer valor contendo 'expression' (case-insensitive check secundário além da regex de formato); validado por payload P5 no corpus |
| T-03-05 | Tampering | CSS Injection via url(javascript:) polyglot | mitigate | sanitizeCssColor() rejeita 'url' keyword no check secundário; validado por payload P6 no corpus |
| T-03-06 | Tampering | HTML attribute break-out (P8: " onmouseover=) | mitigate | outputEscape:'escape' converte " → &quot; em campos text/image — não há escape de atributo manual necessário; validado por payload P8 no corpus |
| T-03-07 | Tampering | Null-byte injection (P10: \x00<script>) | mitigate | sanitize-html parseia o HTML e descarta o null-byte como parte do parser HTML (não interpreta como tag); outputEscape escapa < para &lt; em text; validado por payload P10 no corpus |
| T-03-08 | Tampering | allowProtocolRelative:false ausente em sanitize-html config | mitigate | RICHTEXT_SANITIZE_OPTIONS inclui `allowProtocolRelative: false` explicitamente — rejeita //example.com que poderia virar http: |
| T-03-09 | Elevation of Privilege | SSTI via prototype chain (P1, P2) em valores text | mitigate | Valores text passam como strings de dados para o scope LiquidJS; ownPropertyOnly:true bloqueia acesso a prototype; outputEscape escapa os {{ }} literais se aparecerem no valor (o LiquidJS não re-parseia valores do scope como templates) |
| T-03-10 | Tampering | Bypass de sanitizeCssColor via regex evasion | mitigate | Regex CSS_COLOR_PATTERN usa allowlist (começa com ^, termina com $) — qualquer formato não listado é rejeitado; bypass por inserção de conteúdo fora do padrão é estruturalmente impossível com regex anchored |
| T-03-11 | Tampering | XSS via payload injetado dentro de item de repeater | mitigate | O pipeline de sanitização se aplica no pre-processamento de renderer.ts antes do render; campos dentro de repeater são processados recursivamente da mesma forma que campos diretos; validado pelo 6º contexto 'repeater-text' no corpus (10 payloads adicionais) |
