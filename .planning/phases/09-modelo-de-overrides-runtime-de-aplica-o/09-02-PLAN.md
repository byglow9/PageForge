<objective>
Build the runtime override apply shim and wire it into both the serve route and the export route VITE_SPA branch, achieving the preview==export guarantee.

The shim is a self-contained inline `<script>` that runs after DOMContentLoaded and applies each override by DOM path (text via textContent, color via CSS custom property). The overrides JSON is embedded in the HTML as a JSON-encoded data blob (not a script variable to avoid eval); the shim reads it from a data attribute on a sentinel `<script>` tag.

Purpose: After Plan 01 established the data model, this plan closes the loop from storage to runtime — a seeded override becomes visible in the LP without any UI interaction.

Output: New `apps/web/src/lib/overrides/` module; patched serve + export routes.
</objective>

Create apps/web/src/lib/overrides/apply-shim.ts with two exported functions:

1. buildOverrideInjection(values: ViteSpaValues | null | undefined): { shimScript: string; overridesJson: string }

   - If values is null/undefined OR values.overrides is empty (length === 0): return { shimScript: '', overridesJson: '' }.
   - Otherwise:
     a. overridesJson: build a `<script>` sentinel tag:
        `<script id="pf-overrides" type="application/json">${JSON.stringify(values.overrides)}</script>`
        JSON.stringify provides XSS safety: < > & characters are unicode-escaped by JS engines
        (or use a simple replacer: s.replace(/</g, '\\u003c').replace(/>/g, '\\u003e').replace(/&/g, '\\u0026'))
        to ensure the JSON blob cannot break out of the script tag context.
     b. shimScript: build an inline `<script>` (NOT defer/async — runs synchronously after </head>
        but still waits for DOMContentLoaded for DOM availability):
        The script must:
        - Read the JSON from document.getElementById('pf-overrides').textContent
        - Parse it as JSON (in a try/catch — malformed JSON must not crash the page)
        - For each override in the array (on DOMContentLoaded):
          * If type === 'text': walk the path (split '/' filter empty → childNodes index chain from document.body); set node.textContent = value (NEVER innerHTML)
          * If type === 'color': document.documentElement.style.setProperty('--primary', hexToHslTriplet(value)) where hexToHslTriplet is the same algorithm as theme.ts (inline the pure function in the script string to keep the shim self-contained; it does NOT import from theme.ts — the shim runs in the browser, not Node)
          * If type is unknown or DOM path traversal fails: silently skip (try/catch per override)
        - Path traversal helper: pathToNode(path) walks document.body.childNodes by index (path '/0/2/1' → childNodes[0].childNodes[2].childNodes[1]); returns null if any step fails
        - The entire DOMContentLoaded handler is wrapped in a try/catch that swallows errors

   The shimScript is returned as a raw `<script>` HTML string (the route handler injects it verbatim before </head>).

2. injectOverrides(html: string, injection: { shimScript: string; overridesJson: string }): string

   - If both shimScript and overridesJson are empty, return html unchanged.
   - Insert both tags before </head>: first the overridesJson sentinel, then the shimScript.
   - Use the same pattern as injectBrandStyle: find html.indexOf('</head>'), slice and insert.
   - Fallback (no </head>): prepend to html.

Import PfOverride, ViteSpaValues from '@/lib/lps/schema'.

Security notes to include as JSDoc:
- textContent (not innerHTML) — prevents XSS from override values
- JSON.stringify with unicode escaping — prevents script tag breakout
- try/catch per override — prevents malformed path from crashing LP
- color value in the shim uses the same hex→HSL conversion (pure math, no eval)

Create apps/web/src/lib/overrides/apply-shim.test.ts with all behavior cases above.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Postgres LandingPage.values → serve route HTML | Override values from DB are embedded into HTML served to browsers |
| Inline shim script → LP DOM | The shim walks DOM by path and sets textContent; no eval, no innerHTML |
| Exported ZIP index.html → end user browser | Same injection as serve; ZIP is offline — no server-side CSP header available |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-02-01 | Injection (XSS) | apply-shim.ts — overridesJson embedded in HTML | mitigate | JSON.stringify with unicode-escaping of < > & ensures the JSON blob cannot close the `<script type="application/json">` tag; shim reads via textContent of the sentinel element |
| T-09-02-02 | Injection (XSS) | Shim applies value via node.textContent | accept | textContent never interprets HTML; stored XSS via override value is fully neutralized — this is the explicit design decision per CONTEXT.md |
| T-09-02-03 | Injection (CSS) | color override → CSS setProperty | mitigate | primaryColorOverride validated as /^#[0-9a-fA-F]{6}$/ by SaveViteSpaOverridesSchema before persisting; hexToHslTriplet output is digits + '%' + spaces only — no CSS injection vector |
| T-09-02-04 | Information Disclosure | serve route LP lookup — cross-tenant values read | mitigate | servingRead helper sets app.serving = 'on' RLS flag; landingPage.findFirst scoped to workspaceId derived from HMAC token claims (not client input) |
| T-09-02-05 | Tampering | Shim DOM path traversal — malformed path crashes LP | mitigate | Each override application is wrapped in try/catch; exception silently skipped — LP renders normally without override rather than crashing |
| T-09-02-06 | Denial of Service | Large overrides array embedded in HTML | accept | Array length unbounded in Phase 9; Phase 12 will cap at reasonable limit. Risk is low: LP owner controls their own overrides; no cross-tenant risk |
</threat_model>
