<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client browser → CatalogGrid | selectedFolderId, searchQuery, activeTagId are pure client-side UI state — no security boundary |
| Client browser → setTagsForLpAction | tagNames array is untrusted user input sent to server |
| RSC page → listAllLpTagsForWorkspaceAction | workspaceId from session; all LP/tag data filtered by workspaceId |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-02-01 | Stored XSS | CatalogFilterBar — tag names rendered as pill text | mitigate | Tag names rendered as React text nodes inside Badge (not dangerouslySetInnerHTML); React escapes all text by default; sanitize-html not needed here |
| T-05-02-02 | Stored XSS | TagInput — tag chip display | mitigate | Same as T-05-02-01: React text node rendering; no raw HTML injection |
| T-05-02-03 | IDOR | setTagsForLpAction | mitigate | lpId validated via db.lp.findById (workspaceId filter) before tag assignment; cross-workspace lpId returns null |
| T-05-02-04 | Tampering | setTagsForLpAction tag count/length | mitigate | Zod SetTagsSchema enforces max 10 tags, max 32 chars each (D-07); server-side enforcement regardless of client input |
| T-05-02-05 | Information Disclosure | listAllLpTagsForWorkspaceAction | accept | Tags are workspace-level metadata; all workspace members have lp.read; no PII exposed; scoped to workspaceId from session |
| T-05-02-06 | Spoofing | CatalogFilterBar activeTagId client state | accept | activeTagId drives client-side UI filter only; no server-side query uses it as an untrusted param; the actual tag data is loaded server-side and passed as props |
</threat_model>
